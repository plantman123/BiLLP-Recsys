# 汇报稿


## 拓展实验

接下来，针对原文实现的一些细节方面，我们设计了三个扩展实验，与大家一同探讨：

### 实验一：探究 Reflection Memory 池对实验结果的影响

原论文方法中，Planner 的记忆库会保留所有历史 reflection，随着训练进行，记忆库不断增长。我们想探究：如果限制 reflection memory 的大小，是否会因为保留更高质量的经验而提升效果？还是因为丢失早期经验而降低性能？

方法对比：

| 策略 | 行为 | memory 大小 |
|---|---|---:|
| Baseline (Full) | 保留全部 reflections，不进行裁剪 | 无限制 |
| FIFO | 仅保留最近加入的 N 条 reflections，先进先出 | 15 |
| LRU | 保留最近被检索使用过的 N 条 reflections，按访问热度淘汰 | 15 |

实验参数：

为保证公平对比，三组实验统一使用 episode 检索模式（仅在 episode 开头检索一次 reflection），每次检索 2 条 reflection。初始 memory 相同，包含 50 条 Planner Reflection、320 条 Actor Memory 和 318 条 Critic Memory。实验在 Steam 数据集上进行，共 50 个用户，Max_Iteration=50，env_threshold=30。

实验中 FIFO 和 LRU 的 memory 大小均设为 10，裁剪在初始化加载后和每批 episode 产生新 reflection 后执行。LRU 的访问记录通过 `_mark_reflections_used()` 在每次检索时更新，因此 LRU 初始阶段行为与 FIFO 接近，随着 episode 推进才逐渐体现按热度保留的特点。

实验结果：

| 方法 | 用户数 | Len | R_each | R_traj | Stop Rate |
|---|---:|---:|---:|---:| ---:|
| Baseline | 50 | 4.48 | 4.3795 | 19.62 | 100% |
| FIFO | 50 | **4.68** | 4.3761 | **20.48** | 100% |
| LRU | 50 | 4.60 | **4.3870** | 20.18 | 100% |

在当前实验中，FIFO 策略取得了最好的长期推荐效果，R_traj（20.48）和 Len（4.68）均高于 Baseline（19.62 和 4.48）。LRU 的表现介于两者之间，R_traj 为 20.18，Len 为 4.60。两种有限 memory 策略的长期指标均优于保留全部 reflections 的 Baseline。

这可能有以下原因：一是随着实验进行，早期生成的 reflection 质量参差不齐，保留全部可能引入噪声，有限的 memory 起到了"优胜劣汰"的效果；二是当前实验仅包含 50 个用户、50 轮交互，reflection 总量尚未积累到足以体现 Full memory 长期多样性的程度。FIFO 略优于 LRU，说明在目前实验规模下，保留最新经验比按访问频次保留更有优势——新生成的 reflection 针对当前批次的用户行为，时效性更强。

后续可在更大用户规模下验证：继续增大 memory 大小（如 20、50），观察是否存在最优 memory 容量；在更多轮交互后比较 LRU 与 FIFO，看 LRU 的访问热度机制是否能更好地保留长期有效的经验。


### 实验二：探究不同规模的 Grounding 映射对实验结果的影响

原论文的 Grounding 流程中，Actor 生成一个推荐动作后，通过 embedding 相似度在真实物品目录中找到最接近的物品，并直接选择第一个候选。这一过程只依赖向量相似度，没有利用 LLM 的理解能力进行二次判断。我们想探究：如果将 Grounding 返回的 Top-k 候选交给 LLM 做二次筛选，能否提高最终推荐质量？

方法对比：

| 方法 | Grounding 行为 | LLM 二次筛选 |
|---|---:|---:|
| Baseline (Original) | 返回 Top-1 候选，直接选择第一个 | 无 |
| Top-5 Rerank | 返回 Top-5 候选，LLM 从 5 个候选中选择最优 | 有 |
| Top-10 Rerank | 返回 Top-10 候选，LLM 从 10 个候选中选择最优 | 有 |

实验参数：

三组实验统一使用 original 检索模式（与原论文一致）和 Full memory 策略。实验在 Steam 数据集上进行，共 50 个用户，Max_Iteration=30，env_threshold=30，Max_Reflections=2。二次筛选时，LLM 会结合用户历史、当前推荐轨迹、reflection 和 actor memory 进行综合判断，从候选列表中选择一个最合适的物品。若 LLM 输出无法匹配候选列表，则回退到第一个候选（fallback 机制）。

需要注意的是，Top-k 候选列表不会写入 Actor scratchpad，仅保存在结果 JSON 的 `grounding_reranks` 字段中，因此 rerank 版本不会在后续推荐轮次中看到额外信息。

实验结果：

| 方法 | 用户数 | Len | R_each | R_traj | Stop Rate | Invalid Rate |
|---|---:|---:|---:|---:|---:|---:|
| Baseline | 50 | 4.82 | **4.4315** | 21.36 | 100% | 0% |
| top-5 | 50 | **5.60** | 4.3286 | **24.24** | 98% | 2% |
| top-10 | 50 | 5.10 | 4.4450 | 23.24 | 98% | 2% |

在当前实验中，Top-5 Rerank 取得了最好的长期推荐效果，R_traj（24.24）相比 Baseline（21.36）提升了约 13.5%，Len（5.60）相比 Baseline（4.82）提升了约 16.2%。这说明让 LLM 在 Grounding 返回的多个候选中进行二次筛选，能够选出更符合用户长期兴趣的物品，从而延长推荐轨迹、提高累计收益。

Top-10 Rerank 的表现介于两者之间，R_traj 为 23.24，Len 为 5.10，仍优于 Baseline 但不如 Top-5。这可能是由于候选数量增多后，LLM 的选择难度增加，候选之间的区分度降低，导致二次筛选的效果不如 Top-5 集中。

值得注意的是，Top-5 的 R_each（4.3286）低于 Baseline（4.4315），但 R_traj 更高。这说明 LLM 二次筛选更倾向于选择有利于长期留存但单轮奖励略低的物品，体现了一定的长期规划能力。此外，Rerank 组出现了少量 Invalid（2%）和 Stop（2%），说明二次筛选增加了额外的 LLM 调用，引入了一定的不稳定性。

从成本角度来看，Top-5 Rerank 每轮推荐需要额外一次 LLM 调用来完成二次筛选，增加了推理开销。后续可以进一步研究：不同候选数量（如 Top-3、Top-7）的最优设置；更高效的融合方式，如将候选信息直接整合到 Actor 的推荐 prompt 中，避免额外的 LLM 调用。



### 实验三：探究 Planner Reflection 的检索时机对实验结果的影响

原论文方法中，Planner 在每轮推荐时检索 reflection，但检索时机仅在 episode 开始时执行一次，使用用户的初始历史状态 $s_1$ 作为 query。然而，推荐是一个多轮交互过程——随着 episode 推进，系统已经为用户推荐了多个物品，用户的当前状态 $s_n$ 已经发生了变化。初始状态的 reflection 是否能覆盖推荐过程中的需求？每一步根据最新状态动态检索 reflection 是否能带来更好的长期效果？本实验围绕这一问题展开。

方法对比：

| 方法 | Episode 开头检索 | 每一步动态检索 | 每步使用数量 | 检索说明 |
|---|---:|---:|---:|---|
| Episode | 2 条 | 0 条 | 2 条 | 用初始状态 $s_1$ 检索一次，后续步骤复用，Baseline |
| Dynamic | 0 条 | 2 条 | 2 条 | 每一步用当前状态 $s_n$ 重新检索 |
| Hybrid | 1 条 | 1 条 | 2 条 | 同时使用 $s_1$ 的静态 reflection 和 $s_n$ 的动态 reflection |

三种模式的状态定义：

- **$s_1$（Episode 查询）**：用户进入 episode 前的历史物品列表，即原论文的 `self.task[id]`
- **$s_n$（Dynamic 查询）**：初始历史物品 + 当前 episode 已推荐过的物品，保留最近 `reflection_query_window=15` 个物品

查询语句格式为："The user's viewing history is [...], please recommend item for this user"，Dynamic 模式使用相同的句式，仅更新物品列表。

实验参数：

三组实验统一使用 Full memory 策略（不限制 reflection 数量），关闭 Grounding rerank，`reflection_query_window=15`。为保证每步 Planner 看到的 reflection 总数一致，Episode 和 Dynamic 各使用 2 条，Hybrid 使用静态 1 条 + 动态 1 条。实验在 Steam 数据集上进行，共 10 个用户，Max_Iteration=70，env_threshold=50，Max_Reflections=2。初始 memory 包含 50 条 Planner Reflection、320 条 Actor Memory 和 318 条 Critic Memory。

实现细节上，Dynamic 模式在每一步的 `step()` 方法中调用 `_refresh_step_reflections()`，根据当前已推荐物品列表构建新的 query，通过 FAISS 检索最新的 reflection。Hybrid 模式的动态检索会调用 `_latest_retrieval_selection()` 获取静态检索已选中的 reflection，通过 `exclude_reflections` 参数在动态检索时排除这些内容，避免同一条 reflection 在 Planner prompt 中出现两次。

实验结果：

| 方法 | 用户数 | Len | R_each | R_traj | Stop Rate | Invalid Rate |
|---|---:|---:|---:|---:|---:|---:|
| Episode | 10 | 4.80 | **4.7500** | 22.80 | 90% | 10% |
| Dynamic | 10 | **5.10** | 4.5686 | **23.30** | 90% | 10% |
| Hybrid | 10 | 3.30 | 4.7273 | 15.60 | 80% | 20% |

在当前实验中，Dynamic Retrieval 取得了最好的长期推荐效果，R_traj（23.30）略高于 Episode（22.80），Len（5.10）也优于 Episode（4.80）。这说明随着用户状态动态检索 reflection 是有价值的——推荐过程中用户已经接触了新的物品，基于当前状态的 query 能够检索到与当前阶段更相关的历史经验，从而帮助 Planner 做出更适应当前情境的规划。

Episode Retrieval 的 R_each（4.7500）在三组中最高，高于 Dynamic（4.5686）。这意味着仅使用初始状态的 reflection 时，Planner 更倾向于推荐单轮满意度高的物品；而 Dynamic 模式虽然单轮略低，但通过更贴合当前状态的规划延长了轨迹（Len 更高），最终获得了更高的累计收益。这与实验二的现象类似——牺牲部分即时奖励换取了更好的长期效果。

Hybrid 方法的表现不如预期，R_traj（15.60）明显低于前两组，Len（3.30）也大幅下降，Stop Rate（80%）和 Invalid Rate（20%）与其他两组相比各有优劣。简单将静态和动态 reflection 拼接使用并未产生协同效果。分析可能原因：一是静态和动态 reflection 可能存在信息冲突——基于 $s_1$ 检索的经验建议与基于 $s_n$ 检索的建议方向不一致，导致 Planner 的决策出现摇摆；二是每次仅使用 1 条静态 + 1 条动态 reflection，单条 reflection 覆盖的信息量不足，削弱了各自的作用；三是 Hybrid 模式使 prompt 结构更复杂，可能对 LLM 的推理产生干扰。后续需要研究更合理的融合方式，例如根据推荐阶段动态调整静态与动态 reflection 的权重，或通过 LLM 对两类 reflection 进行摘要整合后再输入 Planner。

本实验的主要局限在于用户规模较小（仅 10 个），Dynamic 相比 Episode 的提升幅度也不大（R_traj 提升约 2.2%）。后续需要扩大用户规模以验证结论的统计显著性，并测试不同 `reflection_query_window` 大小对 Dynamic 模式效果的影响。





总结一下，这篇论文的核心思想是：

**长期推荐不能只看当前点击，还要考虑多轮交互后的用户留存和累计收益。传统强化学习在推荐数据稀疏时容易不稳定，而 LLM 具备一定规划能力。BiLLP 通过 Planner、Reflector、Actor、Critic 四个模块，让 LLM 从历史推荐轨迹中学习宏观策略和微观动作，从而提升长期推荐效果。**
